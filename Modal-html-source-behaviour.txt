Modal Overlay Behaviour Analysis
=================================
Source: 5 full HTML dumps from MyOshi profile pages (SakuyoiVT & LaceSpades)
Purpose: Understand what causes scroll lock when modals open


KEY FINDING
===========
Platform JS sets `overflow: hidden` on <body> whenever a .modal-overlay exists in the DOM.
This is a 1:1 correlation — modal present = scroll locked, modal absent = scroll free.
Cannot be overridden from custom CSS because body is an ANCESTOR of .profile-page.profile-custom-css.


DOM STRUCTURE (consistent across all scenarios)
===============================================
<body style="...">
  <header class="header">...</header>
  <main>
    <div class="profile-page profile-custom-css theme-dark is-desktop">
      <style>...custom CSS...</style>
      <div class="container">...profile content...</div>
      <div class="modal-overlay">...modal...</div>      <!-- only when modal is open -->
    </div>
  </main>
  <footer>...</footer>
  <nav class="mobile-nav">...</nav>
  <section aria-label="Notifications alt+T">...</section>  <!-- sonner toast portal, always empty -->
  <next-route-announcer>...</next-route-announcer>
  <protonpass-root-XXXX>...</protonpass-root-XXXX>         <!-- browser extension, irrelevant -->

CONSISTENT PATTERNS
===================
- .modal-overlay is always a SIBLING of .container, both inside .profile-page
- .modal-content always has style="max-width: 320px;"
- .modal-content always has class="card" with .card-header and .card-body
- No modals render in a portal <section> outside .profile-page (the sonner section is always empty)
- The sonner toast <section aria-label="Notifications alt+T"> exists in all scenarios but is always empty


IMPLICATIONS FOR CUSTOM CSS
============================
1. body { overflow: auto !important; } — CANNOT WORK
   Gets prefixed to: .profile-page.profile-custom-css body { ... }
   body is not a descendant of .profile-page, so this selector matches nothing.

2. .container:has(~ .modal-overlay) — WORKS for detecting modal open state from within profile CSS
   Both .container and .modal-overlay are children of .profile-page

3. .modal-content[style*="320"] — WORKS for targeting the dialog box itself

4. The scroll lock is platform JS behavior, not CSS. It cannot be overridden from custom CSS scope.
   Would require either:
   - Platform to not lock scroll for these small confirmation dialogs
   - A userscript/extension to remove overflow:hidden from body
   - Accepting the scroll lock as unavoidable
